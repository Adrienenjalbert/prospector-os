Why Manual First?

Core Principle: If you can't do it manually in 30 minutes, you can't spec it for automation. Every AI workflow we build must be modelled on a human process we've already validated.

The 2-week audit phase exists to answer three questions before writing a single line of automation:

🔵 Process

What actually happens today?

  • Map as-is workflows step by step
  • Identify where time is wasted
  • Find the "golden path" vs. edge cases
  • Document who does what, when, and why

🟡 Data

What data exists and can we access it?

  • Audit every data source needed
  • Check access, freshness, quality
  • Identify gaps before they block a build
  • Map cross-system joins required

🟢 Outcome

Can we create useful output by hand?

  • Manually create 3+ real outputs per initiative
  • Test with actual stakeholders for feedback
  • Validate that the output is worth automating
  • Capture the "golden example" for AI to replicate
Anti-pattern to avoid: Building a system that produces beautiful outputs nobody asked for. The manual phase forces you to validate demand before investing in automation.

The Three Audit Pillars — Detailed

🔵 PROCESS Audit

For each initiative, you are documenting the current human workflow. The goal is a step-by-step map that an AI system can eventually replicate or accelerate.

What to capture

  • Trigger: What starts this workflow? (New lead, call completed, QBR scheduled, etc.)
  • Steps: Walk through each action — what systems are opened, what data is pulled, what document is created
  • Time: How long does each step take? Where are the bottlenecks?
  • People: Who is involved? Who hands off to whom?
  • Exceptions: What happens when the standard process breaks? What edge cases exist?
  • Output: What's the deliverable at the end? Who receives it? In what format?

How to capture it

Shadow real users doing real work. Don't ask them to describe their process — watch them do it. People often skip steps they've automated in their heads. Sit with Brett during pre-call prep. Sit with an AD during QBR prep. Sit with a CSM during account review.

🟡 DATA Audit

For each initiative, you are verifying that the data the AI system would need actually exists, is accessible, and is trustworthy.

Data audit checklist per source

QuestionWhy It Matters
Does the data exist?Sometimes the field exists but is never populated
Can we access it programmatically?Dashboard access ≠ API access. Check for MCP/API availability
How fresh is the data?Real-time vs. daily sync vs. weekly export — each has implications
What's the completeness rate?If 40% of records are missing key fields, the AI output will be unreliable
Who owns this data source?No owner = no one to fix issues when they arise
Are there access/permission blockers?IT approvals, licence costs, data governance policies

🟢 OUTCOME Validation

For each initiative, you are manually creating the exact output the AI system would produce — using real data, for real stakeholders, and collecting real feedback.

Outcome validation criteria

CriterionWhat "Pass" Looks Like
UsefulnessStakeholder would use this output in their workflow without significant changes
AccuracyData points are correct and current; no misleading information
TimelinessOutput can be produced fast enough to be useful (pre-call, pre-QBR, etc.)
FormatOutput format (Slack, doc, deck) matches how stakeholders actually consume information
RepeatabilityThe process for creating this output can be described clearly enough to automate

Initiative Readiness Scorecard

At the end of the audit, score each initiative on four dimensions. This drives the Go/No-Go decision and confirms or adjusts the build sequence.

Dimension 1 — Not Ready 3 — Partially Ready 5 — Ready
Process Clarity Workflow unclear, many edge cases, no consensus on "right way" Core workflow mapped but edge cases unresolved Clear, repeatable workflow documented and validated by users
Data Availability Key data sources missing, inaccessible, or unreliable Most data available but some gaps or quality issues All required data accessible, fresh, and quality-verified
Output Quality Manual outputs rejected or heavily modified by stakeholders Outputs useful but need significant iteration Stakeholders would use outputs as-is in their workflow
Stakeholder Buy-in No clear champion, low enthusiasm, unclear value Champion identified but broader team uncertain Active champion + team excited to use automated version
Decision rule: Initiatives scoring ≥16/20 proceed to Sprint 1 build. Initiatives scoring 12-15 proceed with caveats (documented risks). Initiatives scoring <12 need additional groundwork before building.

Decision criteria

SignalGoConditional GoNo-Go
Scorecard total ≥16/20 12-15/20 <12/20
Data availability All sources accessible Core sources accessible, gaps have workarounds Critical data missing with no path to access
Stakeholder feedback "I'd use this every day" "Useful but needs work" "Not sure this helps"

After Go decision: Manual outputs become the "golden examples" and test cases for Sprint 1. Process maps become the automation specs. Data audit becomes the connector requirements. Nothing is thrown away — every piece of audit work feeds directly into the build.
